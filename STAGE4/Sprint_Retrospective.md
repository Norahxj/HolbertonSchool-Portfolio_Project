# Sprint Retrospective

## Purpose

The Sprint Retrospective was conducted at the end of each sprint to reflect on the team's performance, identify successful practices, discuss challenges, and define improvements for future sprints.

---

# Sprint 1 Retrospective

## What went well?

* The backend architecture was successfully established.
* Authentication and child management features were implemented as planned.
* Team communication remained effective through regular meetings.
* GitHub collaboration and version control worked smoothly.
* API documentation supported frontend preparation.

## What challenges did we face?

* Setting up the project structure required additional coordination.
* Some API endpoints required refinement after initial implementation.
* Preparing the frontend while the backend was still under development required close collaboration.

## What can we improve next time?

* Finalize API specifications earlier.
* Increase code reviews before merging pull requests.
* Improve sprint estimation for backend development.
* Continue maintaining clear communication between backend and frontend teams.

---

# Sprint 2 Retrospective

## What went well?

* Task management features were successfully implemented.
* Frontend screens were integrated with backend APIs.
* Team members collaborated effectively during API integration.
* Continuous testing helped identify issues early.

## What challenges did we face?

* Some API integrations required additional debugging.
* A few UI components needed redesign after integration.
* Minor synchronization issues occurred between frontend and backend development.

## What can we improve next time?

* Perform integration testing throughout the sprint instead of waiting until the end.
* Improve API documentation with additional examples.
* Continue using feature branches for isolated development.
* Allocate additional time for frontend validation.

---

# Sprint 3 Retrospective

## What went well?

* The parent approval workflow was completed successfully.
* Noor Points calculation and updates worked as expected.
* Parent Review and Points Dashboard were integrated without major issues.
* Team collaboration remained consistent throughout the sprint.

## What challenges did we face?

* Testing approval and rejection scenarios required multiple iterations.
* Ensuring accurate point updates across all workflows required additional validation.
* Some edge cases were identified during integration testing.

## What can we improve next time?

* Expand automated testing for approval workflows.
* Improve validation for task status changes.
* Increase regression testing before sprint completion.
* Continue improving communication during integration.

---

# Sprint 4 Retrospective

## What went well?

* Wishlist and Weekly Rewards features were completed successfully.
* Final integration testing confirmed the stability of the application.
* Most reported bugs were resolved before project completion.
* The MVP met its planned objectives and was ready for demonstration.

## What challenges did we face?

* Final testing uncovered several minor bugs that required immediate fixes.
* Performance optimization required additional development time.
* Coordinating final verification across all modules required careful planning.

## What can we improve next time?

* Begin performance optimization earlier in the project.
* Allocate additional time for final regression testing.
* Improve sprint planning for optional features.
* Continue refining the development and testing workflow based on lessons learned.

---

# Overall Retrospective Summary

Throughout the four sprints, the team continuously improved collaboration, development practices, testing processes, and project organization. Regular communication, code reviews, and iterative development enabled the successful delivery of the MVP while identifying valuable improvements for future project iterations.
